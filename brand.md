## Palette — ChatGPT Monochrome

**Vibe:** monochrome · neutral · no accent color
**Category:** ai / tech
**Mood:** technical · premium (initial candidates), superseded by an explicit user request to match ChatGPT's own color scheme

The initial brand interview generated 6 candidates in the "technical + premium" direction (Quantum Lab, Meridian, Bone White, Violet Circuit, Ultraviolet, Slate Signal — all blue-violet accent palettes). The user reviewed the preview and asked for ChatGPT's actual look instead: a deliberately colorless, high-contrast neutral scheme. That reference was already named in the brand interview's "reference brands" question, so this isn't a deviation from the process — it's the process's own reference-brand step taking priority over the generated candidates.

## Typography

Kept the project's existing pairing: **Geist** (sans) + **Geist Mono** (mono), already wired via `next/font/google` in `app/layout.tsx` since Phase 0. Not re-run through the typography preview — Geist is already a clean, geometric-humanist sans very close in spirit to ChatGPT's own typeface, so switching would have been low-value churn given how decisively the color direction was chosen. Revisit with `/brand-design` if a different pairing is ever wanted.

## Gradients

Not used, and not appropriate here — the entire point of this palette is the absence of color. Adding a gradient would directly fight the "monochrome, content is the focus" intent that was the actual reason for choosing it.

## Tone and voice

**Words to use:** direct, specific, factual — "answers are grounded in and cited from your files," not "unlock the power of your documents." The product's whole value proposition is trustworthiness (no hallucination), so copy should mirror that: plain claims, no hype.

**Words to avoid:** hype words (revolutionary, supercharge, unleash), emojis in UI copy, exclamation marks, vague claims like "smart" or "powerful" without specifics.

**Voice example:** "Nothing relevant was found in your documents for this question." — states the honest outcome plainly, no apology padding.

## Usage dos and don'ts

**Do:**
- Use the CSS variables (`bg-background`, `text-foreground`, `bg-primary`, `border-border`, `text-muted-foreground`) everywhere — never hardcode a hex or a Tailwind color like `zinc-200`.
- Keep buttons and focus rings on `primary`/`ring` — do not introduce a colored accent; it would break the deliberate monochrome identity.
- Test every component in both light and dark mode (toggle OS appearance, or `prefers-color-scheme` emulation in devtools/Playwright) before considering it done.

**Don't:**
- Add a brand accent color "to make it pop" — that's explicitly the opposite of this palette's intent.
- Reintroduce hardcoded `zinc-*` / `red-*` / `green-*` Tailwind color classes for anything that should track the theme (status badges like "ready"/"failed" are an intentional exception — they're semantic state colors, not brand colors).
- Override these tokens per-component — if a color is wrong, fix it once in `app/globals.css`.
